**CliOptions.cs** — Add `bool Enrich = false` parameter to the record, after DryRun. Update the doc comment to describe the --enrich flag.

**PipelineResult.cs** — Add enrichment metrics properties:
- `int EntitiesEnriched { get; set; }` — number of entities that got new LLM summaries
- `int EntitiesCached { get; set; }` — number of entities served from summary cache
- `int EntitiesFailed { get; set; }` — number of entities where LLM failed
- `int InputTokensUsed { get; set; }` — total input tokens consumed
- `int OutputTokensUsed { get; set; }` — total output tokens consumed
- `int EstimatedTotalTokens { get; set; }` — pre-enrichment estimate for progress display

**Program.cs** — Add enrichment integration:

1. Add `--enrich` CLI option:
   ```csharp
   var enrichOption = new Option<bool>("--enrich")
   {
       Description = "Generate LLM-powered plain-English summaries for methods and classes"
   };
   ```
   Add to rootCommand. Parse in SetAction lambda. Pass to RunPipelineAsync.

2. Update RunPipelineAsync signature to accept `bool enrich` parameter.

3. Add enrichment setup logic BEFORE pipeline execution (after solution load, before running pipeline):
   - If `enrich` is true:
     a. Determine config path: `Path.Combine(Path.GetDirectoryName(solutionPath)!, "code2obsidian.llm.json")`
     b. Try load config: `LlmConfigLoader.TryLoad(configPath)`
     c. If config is null: run `InteractiveSetup.RunSetup(configPath)`. If setup returns null (non-interactive), print error "No LLM configuration found. Create code2obsidian.llm.json or run interactively." and return exit code 1.
     d. Create IChatClient: `ChatClientFactory.CreateFromConfig(config)` — wrap in try/catch, print clear error on failure.
     e. Create SummaryCache: `new SummaryCache(stateDbPath)` where stateDbPath = `Path.Combine(outputDir, ".code2obsidian-state.db")`
     f. Create LlmEnricher: `new LlmEnricher(client, cache, config, progress)` — progress wired later per execution case.

4. Cost estimation and confirmation (before pipeline runs, after analysis):
   This is tricky because we need the AnalysisResult to estimate cost. The approach:
   - For the NON-incremental path (Case A): The cost estimation must happen BETWEEN analysis and enrichment. Since Pipeline.RunAsync runs all three stages, we need to either:
     - Split it: run analysis first, then estimate, then enrich+emit. OR
     - Pre-estimate based on solution size heuristic.
   - RECOMMENDED approach: Add a pre-enrichment hook. Before calling pipeline.RunAsync, if enrich is true:
     - Run analysis-only to get entity count (or estimate from file count)
     - Actually, simpler: let Pipeline.RunAsync run, and have LlmEnricher itself do the cost check at the START of EnrichAsync. This keeps the pipeline architecture clean.
   - BEST approach: Move cost estimation INTO LlmEnricher.EnrichAsync, at the top, before any LLM calls:
     - Count uncached entities (cache lookups are fast)
     - If `CostEstimator.ShouldConfirm(uncachedCount)`:
       - Compute average prompt tokens from a sample (first 5 methods)
       - Call `CostEstimator.EstimateCost`
       - Use `AnsiConsole.Confirm` to prompt — BUT LlmEnricher should not depend on Spectre.Console.
     - SOLUTION: Add a `Func<int, int, int, decimal, bool>? confirmEnrichment` parameter to LlmEnricher constructor. Program.cs provides a lambda that uses AnsiConsole.Confirm. The function receives (uncachedCount, estimatedInputTokens, estimatedOutputTokens, estimatedCost) and returns true to proceed. If the function is null or returns true, enrichment proceeds. If false, skip enrichment (log message, return, leave enriched dictionaries empty).
   - In Program.cs, create the confirmation lambda:
     ```csharp
     Func<int, int, int, decimal, bool> confirmEnrichment = (count, inTokens, outTokens, cost) =>
     {
         AnsiConsole.MarkupLine($"[yellow]Enrichment will process {count} entities[/]");
         AnsiConsole.MarkupLine($"[yellow]Estimated: ~{inTokens + outTokens:N0} tokens (~${cost:F4})[/]");
         return AnsiConsole.Confirm("Proceed with LLM enrichment?", defaultValue: true);
     };
     ```
     Pass this lambda to LlmEnricher constructor.

5. Live token progress during enrichment:
   - LlmEnricher already reports through IProgress<PipelineProgress>.
   - Update the PipelineStage.Enriching progress handler in Program.cs to show token counts.
   - After enrichment completes, read LlmEnricher's `InputTokensUsed`, `OutputTokensUsed`, `EntitiesEnriched`, `EntitiesCached`, `EntitiesFailed` and store on PipelineResult.

6. Wire enricher into pipeline:
   - Non-incremental Case A (RunFullPipeline): if `enrich`, create LlmEnricher and add to enrichers list (instead of empty list). The Pipeline.RunAsync enrichment loop already calls EnrichAsync on all enrichers.
   - Incremental cases: IncrementalPipeline needs to accept optional enrichers. Update IncrementalPipeline to accept `IReadOnlyList<IEnricher>` in its constructor and pass them through when creating the Pipeline internally.

**IncrementalPipeline.cs** — Accept enrichers:
- Add `IReadOnlyList<IEnricher> _enrichers` constructor parameter.
- In `RunFullWithStateSaveAsync`, `RunIncrementalAsync`, pass `_enrichers` when creating Pipeline.
- In Program.cs incremental cases, construct IncrementalPipeline with the enrichers list (empty if no --enrich, list with LlmEnricher if --enrich).

7. Update RenderSummary to show enrichment metrics:
   - If `result.EntitiesEnriched > 0 || result.EntitiesCached > 0`:
     - Change enrichment row from "skipped" to: `"{enriched} enriched, {cached} cached, {failed} failed"`
     - Add a token usage line: `"Tokens: {input + output:N0} ({input:N0} in, {output:N0} out)"`

8. Update the "Enrichment skipped (Phase 1)" message in Pipeline.cs: change to "No enrichers configured" or similar since it's no longer Phase 1.

**ObsidianEmitter.cs** — Inject `## Summary` section when summaries are available.

The emitter already receives `EnrichedResult` in `EmitAsync`. The render methods currently receive `AnalysisResult` only. We need to thread the enriched result (or the summary dictionaries) through to the render methods.

1. Change render methods to accept summary dictionaries:
   - `RenderMethodNote` — add parameter `string? summary` (nullable). If non-null and non-empty, render a `## Summary` section AFTER the danger callouts and BEFORE the method section (signature/call graph). This places the plain-English explanation prominently:
     ```markdown
     ## Summary
     {summary text}

     ```
   - `RenderClassNote` — add parameter `string? summary`. If non-null, render `## Summary` section AFTER the purpose blockquote and source path, BEFORE the type relationships section:
     ```markdown
     ## Summary
     {summary text}

     ```
   - `RenderInterfaceNote` — add parameter `string? summary`. Same placement as class notes.

2. In `EmitAsync`, when iterating methods:
   - Look up summary: `enrichedResult.MethodSummaries.TryGetValue(methodId.Value, out var summary) ? summary : null`
   - Pass to `RenderMethodNote`

3. In `EmitAsync`, when iterating types:
   - Look up summary: `enrichedResult.TypeSummaries.TryGetValue(typeId.Value, out var summary) ? summary : null`
   - Pass to class/interface render methods

4. CRITICAL for LLM-04: When `--enrich` is not used, `EnrichedResult.MethodSummaries` and `TypeSummaries` are empty dictionaries. The `TryGetValue` calls return false, `summary` is null, and no `## Summary` section is rendered. The notes are identical to pre-Phase-5 output. This requires NO special handling — just the null check on the summary parameter.

5. Keep the existing `EnrichedResult result` parameter in `EmitAsync` (it's already there). Access `result.MethodSummaries` and `result.TypeSummaries` directly in the emission loops.

6. Update `RenderMethodSection` — the self-contained method section render does NOT get a summary (it's a shared helper used by RenderMethodNote). The summary goes in RenderMethodNote before calling RenderMethodSection.

NOTE: The `## Summary` section must have the `##` heading to be consistent with other sections in the notes (## Dependencies, ## Members, ## Properties, ## Known Implementors). This also makes it Dataview-queryable.
