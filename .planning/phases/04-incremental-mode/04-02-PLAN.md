  <action>
All three tables needed by these components (type_metadata, method_index, type_index) are already created in Plan 04-01's StateSchema with corresponding read/write methods on IncrementalState. This task consumes those APIs -- it does NOT modify the schema or state classes.

1. Create `Incremental/RippleCalculator.cs`:
   - Sealed class with static method: `HashSet<string> ComputeAffectedFiles(IReadOnlySet<string> changedFiles, AnalysisResult freshAnalysis, IncrementalState state)`
   - Start with all changed files in the affected set (use StringComparer.OrdinalIgnoreCase).
   - **One-hop callers** (unchanged code calling into changed code): For each method in freshAnalysis whose FilePath is in changedFiles, look up callers in state.GetCallers(methodId.Value). For each stored caller, get its file via state.GetFileForMethod(callerId). Add caller files to affected set.
   - **One-hop callees** (changed code calling into unchanged code): For each method in freshAnalysis whose FilePath is in changedFiles, iterate its CallGraph.GetCallees(). For each callee, get its file from state.GetFileForMethod(calleeId.Value). Add callee files to affected set.
   - **Structural change detection**: Add a separate static method `HashSet<string> DetectStructuralRipple(AnalysisResult freshAnalysis, IReadOnlySet<string> changedFiles, IncrementalState state)`:
     - Load stored type info: state.GetTypeReferences(), state.GetTypeFiles(), and state.GetTypeMetadata().
     - For each type in freshAnalysis.Types whose FilePath is in changedFiles:
       - Check if the TypeId exists in stored type_files. If NOT (new type) -- no wide ripple needed.
       - If it exists, compare stored base class, interfaces, namespace from type_metadata against fresh values. If changed, add ALL files from type_references for that TypeId to the affected set.
     - For TypeIds present in stored type_files for changed files but NOT in freshAnalysis.Types (deleted type): add ALL files from type_references for that TypeId to the affected set.
   - Combine one-hop + structural results into the final affected file set.

2. Create `Incremental/StaleNoteDetector.cs`:
   - Sealed class with static method: `IReadOnlyList<string> FindStaleNotes(IReadOnlyDictionary<string, (string SourceFile, string EntityId)> storedNotes, IReadOnlySet<string> currentEntityIds, IReadOnlySet<string> reanalyzedFiles)`
   - A stored note is stale if:
     (a) Its source_file was reanalyzed (meaning we have fresh data for it), AND
     (b) Its entity_id is NOT in the current emission set (meaning the entity was deleted or renamed).
   - Do NOT mark notes as stale if their source_file was NOT reanalyzed (those files were unchanged -- their notes are still valid).
   - Return the list of note file paths (absolute vault paths) that should be deleted.

3. Create `Incremental/AnalysisResultMerger.cs`:
   - Sealed class with static method: `AnalysisResult Merge(AnalysisResult freshResult, IncrementalState state, IReadOnlySet<string> reanalyzedFiles)`
   - Purpose: Combine freshly analyzed data (for changed + ripple files) with stored data (for unchanged files) into a complete AnalysisResult that the emitter can use as if it were a full analysis.
   - **Methods**: Start with all methods from freshResult. Then load stored method_index via state.GetMethodIndex() for methods whose files are NOT in reanalyzedFiles. Use Approach B (lightweight): method_index only stores (method_id, containing_type, file_path) -- this is sufficient for the emitter's collision detection (wikilinks). The emitter only writes notes for dirty files, so unchanged method metadata is only needed for ContainingTypeName disambiguation.
   - **Types**: Start with freshResult.Types. Load stored type data from state.GetTypeIndex() for files NOT in reanalyzedFiles. type_index stores (type_id, name, full_name, file_path, kind) -- enough for collision detection.
   - **CallGraph**: Merge fresh call graph edges with stored edges from state.GetCallEdges(). Only include stored edges where BOTH caller and callee files are NOT in reanalyzedFiles -- edges involving reanalyzed files come from fresh data.
   - **Implementors**: Reconstruct from stored type_references + fresh data.
   - **Counts**: ProjectCount and FileCount from the full solution (these come from the pipeline, not the merger).
   - The merger returns a complete AnalysisResult usable by the emitter.
  </action>

1. Modify `Analysis/Analyzers/MethodAnalyzer.cs`:
   - Add an optional `IReadOnlySet<string>?` fileFilter field set via constructor parameter (default null).
   - In the document iteration loop (the `foreach (var document in project.Documents)` block), AFTER the null/whitespace FilePath check, add:
     ```
     if (fileFilter is not null && !fileFilter.Contains(document.FilePath!))
     {
         // Skip unchanged files in incremental mode
         continue;
     }
     ```
   - The file filter uses case-insensitive comparison. Ensure the HashSet passed in uses StringComparer.OrdinalIgnoreCase.
   - The `IncrementFileCount()` call should ONLY count files that are actually analyzed (move it after the filter check).
   - Keep the existing constructor as a parameterless overload (fileFilter = null = full analysis mode, backward compatible).

2. Modify `Analysis/Analyzers/TypeAnalyzer.cs`:
   - Same pattern as MethodAnalyzer: add optional fileFilter constructor parameter.
   - Apply the same document-level filter in the iteration loop.
   - Preserve backward-compatible parameterless constructor.

3. Modify `Emission/IEmitter.cs`:
   - Add a new overload to the interface:
     `Task<EmitResult> EmitAsync(EnrichedResult result, string outputDirectory, IReadOnlySet<string>? dirtyFiles, CancellationToken ct)`
   - The existing `EmitAsync(EnrichedResult, string, CancellationToken)` remains as the default (full emission). Add a default interface method that delegates to the new overload with dirtyFiles = null.
   - Alternative (simpler, preferred): Change the existing signature to add the optional parameter with default null:
     `Task<EmitResult> EmitAsync(EnrichedResult result, string outputDirectory, CancellationToken ct, IReadOnlySet<string>? dirtyFiles = null)`
     This is source-compatible since existing callers don't pass dirtyFiles.
     Actually, since CancellationToken is not last, use this approach instead: keep original signature, add new method. Or better yet: just modify ObsidianEmitter to accept dirtyFiles in constructor (like fanInThreshold). This avoids interface changes.
   - DECISION: Add `IReadOnlySet<string>? dirtyFiles` as a constructor parameter on ObsidianEmitter (default null). When null, emit all notes (full mode). When non-null, only emit notes for entities whose FilePath is in the dirty set. The IEmitter interface stays unchanged.

4. Modify `Emission/ObsidianEmitter.cs`:
   - Add `private readonly IReadOnlySet<string>? _dirtyFiles;` field, set via new constructor parameter `IReadOnlySet<string>? dirtyFiles = null`.
   - In the method note emission loop (`foreach (var (methodId, method) in analysis.Methods)`), add at the top:
     ```
     if (_dirtyFiles is not null && !_dirtyFiles.Contains(method.FilePath))
         continue;
     ```
   - In the type note emission loop (`foreach (var (typeId, typeInfo) in analysis.Types)`), add the same check:
     ```
     if (_dirtyFiles is not null && !_dirtyFiles.Contains(typeInfo.FilePath))
         continue;
     ```
   - CRITICAL: The collision detection (BuildCollisionSet, BuildOverloadIndex) must still operate on the FULL AnalysisResult, not just dirty files. This is already the case since those methods iterate analysis.Methods/analysis.Types. The filter only affects which notes are WRITTEN.
   - Track emitted note paths for state storage: collect a `List<(string NotePath, string SourceFile, string EntityId)>` during emission. For method notes: NotePath = the file path written, SourceFile = method.FilePath, EntityId = methodId.Value. For type notes: EntityId = typeId.Value. Expose this via a new property or return it as part of EmitResult.

5. Modify `Emission/EmitResult.cs`:
   - Add a new field: `IReadOnlyList<(string NotePath, string SourceFile, string EntityId)> EmittedNotes` to track what was emitted for state storage.
   - Update the record: `public sealed record EmitResult(int NotesWritten, IReadOnlyList<string> Warnings, IReadOnlyList<(string NotePath, string SourceFile, string EntityId)> EmittedNotes)`
   - Update the constructor call in ObsidianEmitter.EmitAsync to pass the collected emitted notes list.

BACKWARD COMPATIBILITY: All changes use optional parameters with null defaults. Existing code in Pipeline.cs and Program.cs continues to work without modification. The Pipeline.RunAsync still calls `_emitter.EmitAsync(enrichedResult, outputDir, ct)` which works fine -- the emitter just emits everything when dirtyFiles is null.
