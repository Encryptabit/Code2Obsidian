1. Update `Pipeline/Pipeline.cs`:
   - Modify the Pipeline constructor to accept optional `IReadOnlySet<string>? fileFilter` and `IReadOnlySet<string>? dirtyFiles` parameters (both default null).
   - Store as private readonly fields.
   - The Pipeline no longer directly constructs analyzers -- it receives them as-is. The file filter is already baked into the analyzer instances by the caller (Program.cs). So actually, Pipeline does NOT need the fileFilter field. The analyzers already have it.
   - However, Pipeline DOES need to know about dirtyFiles for the emitter. Since the emitter receives dirtyFiles via its constructor (from plan 02), Pipeline also does not need this.
   - REVISED APPROACH: Pipeline.cs stays mostly unchanged. The composition in Program.cs creates the analyzers and emitter with the right filter/dirty parameters. Pipeline just orchestrates as before.
   - One change needed: Pipeline should report the number of files analyzed (which will be fewer in incremental mode). The existing `result.FilesAnalyzed = analysisResult.FileCount` already handles this since MethodAnalyzer only increments file count for analyzed files (per plan 02 change). Good.
   - Add total files count to Pipeline for progress reporting. Accept an optional `int? totalFilesHint` parameter that represents the total files in the solution (for progress display: "Analyzing 12/200 files"). When provided, progress reports use this as the denominator. When null, behavior unchanged.
   - Actually, simpler: just update the progress description string. The analyzers already report progress. In incremental mode, the progress will naturally show fewer files. The "X unchanged" display can be handled in Program.cs when composing the progress adapter.

   FINAL Pipeline.cs changes:
   - No structural changes needed. The existing Pipeline.RunAsync works as-is because:
     (a) Analyzers are pre-configured with file filters (constructed in Program.cs)
     (b) Emitter is pre-configured with dirty files (constructed in Program.cs)
     (c) Progress reporting comes from analyzers naturally
   - The only addition: expose `analysisResult` via a new output or modify PipelineResult to include it, because Program.cs needs the AnalysisResult to save state. Current PipelineResult does not expose the AnalysisResult.
   - Add `AnalysisResult? AnalysisResult { get; set; }` to PipelineResult (nullable to maintain backward compatibility).
   - Add `EmitResult? EmitResult { get; set; }` to PipelineResult to expose emitted notes list for state saving.
   - In Pipeline.RunAsync, set `result.AnalysisResult = analysisResult` and `result.EmitResult = emitResult` before returning.

2. Update `Program.cs` -- full incremental orchestration in RunPipelineAsync:
   The incremental flow is the heart of this plan. The logic in RunPipelineAsync must handle these cases:

   **Case A: No --incremental flag (default)**
   - Behavior unchanged from prior phases. Full analysis, full emission. No state interaction.

   **Case B: --incremental with no prior state (first run / INFR-05)**
   - Try to load state from `.code2obsidian-state.db` in the output directory.
   - State doesn't exist or is corrupt -> IncrementalState.TryLoad returns false.
   - Perform full analysis (no file filter).
   - After successful emission, save state (commit SHA, file hashes, call edges, type refs, type files, emitted notes).
   - Log: "No prior state found. Performing full analysis and saving state."

   **Case C: --incremental with existing state (subsequent runs / INFR-03, INFR-04)**
   - Load state: IncrementalState.TryLoad returns true with commitSha.
   - Detect changes: Try GitChangeDetector first. If it throws (not a git repo), fall back to HashChangeDetector.
   - If no changes detected (ChangeSet.Changes is empty): log "No changes detected. Vault is up to date." and return exit code 0 immediately.
   - **Pass 1 -- Analyze changed files only:**
     - Create MethodAnalyzer and TypeAnalyzer with fileFilter = changeSet.ChangedFilePaths.
     - Run pipeline (analysis only -- but we still run the full pipeline, just with filtered analyzers).
     - Extract fresh AnalysisResult from PipelineResult.
   - **Ripple computation:**
     - Use RippleCalculator.ComputeAffectedFiles(changeSet.ChangedFilePaths, freshAnalysis, state).
     - The result is the full set of affected files (changed + one-hop neighbors + structural ripple).
   - **Pass 2 -- Analyze affected files (if ripple expanded the set):**
     - If affectedFiles is larger than changedFiles:
       - Create new analyzers with fileFilter = affectedFiles.
       - Run analysis again. This produces the complete fresh AnalysisResult for ALL affected files.
     - If affectedFiles == changedFiles: pass 1 result is already complete.
   - **Merge with stored data:**
     - Use AnalysisResultMerger.Merge(freshAnalysis, state, affectedFiles) to get complete AnalysisResult.
   - **Selective emission:**
     - Create ObsidianEmitter with dirtyFiles = affectedFiles.
     - Run emission through pipeline (pass the merged AnalysisResult through EnrichedResult).
   - **Stale note cleanup:**
     - Use StaleNoteDetector.FindStaleNotes(state.GetEmittedNotes(), currentEntityIds, affectedFiles).
     - Delete each stale note file from disk. Count deletions.
     - Log stale deletions only in verbose mode (check if -v was passed -- for now, always log since we don't have -v yet; or just use a comment noting this).
   - **Save state:**
     - After successful emission, save state with fresh data for reanalyzed files + stored data for unchanged files.
     - Extract emitted notes from EmitResult.EmittedNotes. Merge with stored emitted notes for non-reanalyzed files.
     - Call state.SaveState() with the merged data.
   - **Git rename handling:**
     - If changeSet.RenamedPaths has entries: for each old->new rename, check if a note exists at the old path in the vault. If so, rename the note file (preserves Obsidian backlinks if the vault is in git).

   **Case D: --full-rebuild**
   - Call state.DeleteState() to wipe the database.
   - Then perform Case A (full analysis, no filtering).
   - After emission, save fresh state.

   **Case E: --dry-run (with --incremental)**
   - Perform all steps of Case C up to emission, but instead of writing files, just log what WOULD be emitted.
   - Print: "Would regenerate N notes (M from changed files, K from ripple)."
   - Print: "Would delete N stale notes."
   - Print list of affected files.
   - Return exit code 0 without modifying the vault or state.

   **Progress display integration:**
   - In incremental mode, update the Spectre.Console progress description to show skipped counts.
   - Before running the pipeline, calculate total files in solution and files to analyze.
   - Update progress adapter: "Analyzing {analyzeCount}/{totalCount} files ({skipCount} unchanged)"

   **State DB path:** `Path.Combine(outputDir, ".code2obsidian-state.db")`.
   - Add `.code2obsidian-state.db*` to the vault's .gitignore (create one if it doesn't exist, or append).

   **Extracting state data for SaveState:**
   After pipeline completes, extract state data from the AnalysisResult and EmitResult:
   - fileHashes: For each file in the affected set, compute SHA256 hash of the current file content. Merge with stored hashes for non-affected files.
   - callEdges: From AnalysisResult.CallGraph, extract all edges. For each edge, look up the file path from Methods dict. Merge fresh edges (files in affectedSet) with stored edges (files not in affectedSet).
   - typeReferences: From AnalysisResult.Types, for each type, record (typeId, filePath). Also scan Methods for type references via ContainingTypeId.
   - typeFiles: From AnalysisResult.Types, record (typeId, filePath) for all types.
   - emittedNotes: From EmitResult.EmittedNotes, merged with stored notes for non-reanalyzed files.
   - typeMetadata: From AnalysisResult.Types, extract (typeId, baseClass, interfaces, namespace) for structural comparison.
   - methodIndex: From AnalysisResult.Methods, extract (methodId, containingType, filePath).
   - typeIndex: From AnalysisResult.Types, extract (typeId, name, fullName, filePath, kind).
   UPDATE IncrementalState.SaveState to accept all these additional data sets (type_metadata, method_index, type_index). This extends the SaveState signature from plan 01.

   **Implementation note on two-pass:**
   The two-pass approach requires running the pipeline analysis stage twice. To avoid running the full pipeline both times, extract the analysis portion. Options:
   - Option A: Expose AnalysisResultBuilder and call analyzers directly without going through Pipeline. Program.cs would call analyzer.AnalyzeAsync directly, then pass the result to Pipeline for enrichment + emission.
   - Option B: Run Pipeline.RunAsync twice with different analyzers (wasteful -- emitter runs both times).
   - Option C: Split Pipeline into separate Analysis and Emission stages callable independently.
   Use Option A. It is the simplest -- just call analyzer.AnalyzeAsync directly for pass 1, then call Pipeline.RunAsync for the final pass (with merged result). Actually, Pipeline.RunAsync always runs analysis. So:
   - For pass 1: call analyzers directly (MethodAnalyzer.AnalyzeAsync + TypeAnalyzer.AnalyzeAsync with filtered file set).
   - Compute ripple.
   - For pass 2 (if needed): call analyzers directly again with expanded file set.
   - Merge result.
   - For emission: create a modified pipeline run that SKIPS analysis and just does enrichment + emission. OR: bypass Pipeline entirely and call emitter.EmitAsync directly with the merged EnrichedResult.
   The cleanest approach: call analyzers and emitter directly in Program.cs for incremental mode, bypassing Pipeline.RunAsync. Only use Pipeline.RunAsync for full (non-incremental) mode. This avoids modifying Pipeline.cs for the two-pass flow. Pipeline remains the simple sequential orchestrator for full mode.

   FINAL APPROACH:
   - Non-incremental mode: uses existing Pipeline.RunAsync (unchanged).
   - Incremental mode: Program.cs orchestrates directly:
     1. Load solution (same as before).
     2. Detect changes + load state.
     3. Pass 1: call analyzers with changed-file filter.
     4. Compute ripple.
     5. Pass 2 (if needed): call analyzers with expanded filter.
     6. Merge with stored data.
     7. Create EnrichedResult from merged AnalysisResult.
     8. Call emitter.EmitAsync directly with dirty files.
     9. Clean stale notes.
     10. Save state.
   - Extract the Spectre.Console progress wrapping into a helper so both paths use it.
   - PipelineResult is populated manually in incremental mode.
