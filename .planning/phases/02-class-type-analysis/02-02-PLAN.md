**Add to EmitAsync method:**
After the existing method notes loop, add a second loop over `analysis.Types`:

```
foreach (var (typeId, typeInfo) in analysis.Types)
{
    ct.ThrowIfCancellationRequested();
    var fileName = Sanitize($"{typeInfo.FullName}.md");
    var filePath = Path.Combine(outputDirectory, fileName);
    try
    {
        var content = typeInfo.Kind == TypeKindInfo.Interface
            ? RenderInterfaceNote(typeInfo, analysis)
            : RenderClassNote(typeInfo, analysis);
        await File.WriteAllTextAsync(filePath, content, ct);
        notesWritten++;
    }
    catch (IOException ex)
    {
        warnings.Add($"Failed to write '{filePath}': {ex.Message}");
    }
}
```

**RenderClassNote(TypeInfo typeInfo, AnalysisResult analysis) -> string:**

Generate markdown with this structure:

1. **YAML frontmatter** (per user decision: minimal Phase 2 set):
```yaml
---
base_class: "MyNamespace.BaseClassName"    # FullName or ~ for null
interfaces:
  - "MyNamespace.IFoo"
  - "MyNamespace.IBar"
namespace: "MyNamespace"
source_file: "path/to/file.cs"
---
```
Use FullName for base_class and interfaces (matches note file names, collision-safe). Use `~` (YAML null) if base_class is null. Empty list `[]` if no interfaces.

2. **Title**: `# ClassName`

3. **Purpose summary** (STRC-01 requirement):
If `typeInfo.DocComment` is not null, render it as a paragraph immediately after the title:
```
> {DocComment text}
```
Use a blockquote for visual distinction. If DocComment is null, emit a fallback line:
```
> {Kind} in `{Namespace}`
```
This satisfies STRC-01's "purpose summary" requirement — the XML doc `<summary>` provides the purpose when available, and the namespace/kind fallback ensures every note has some descriptive context.

4. **Source path**: `**Path**: \`path/to/file.cs\``

5. **Type relationships** (as wikilinks for user types, plain text for external types):
- If base class exists AND `typeInfo.BaseClassFullName` is not null: check if a type note exists by looking up BaseClassFullName in analysis.Types (by FullName). If found: `**Inherits from**: [[Sanitize(typeInfo.BaseClassFullName)]]`. If not in Types (external): `**Inherits from**: {typeInfo.BaseClassName}` (plain text).
- If interfaces exist: `**Implements**:` followed by bullet list. For each interface, use the corresponding entry from `typeInfo.InterfaceFullNames`. Check if that FullName has a note in analysis.Types. If found: `- [[Sanitize(interfaceFullName)]]`. If external: `- {interfaceName}` (plain text). The wikilink target MUST match the note file name (Sanitize(FullName).md).

**Canonical wikilink rule:** ALL type wikilinks in class/interface notes use `[[Sanitize(FullName)]]` where FullName is the fully qualified type name from the domain model. This is the collision-safe identity that matches note file names.

6. **Dependencies section** (Claude's discretion -- giving DI deps their own section for graph visibility):
Collect DI parameters from ALL explicitly declared constructors (not just the first or largest -- STRC-06 says "constructor parameters are listed", meaning all of them). Deduplicate by TypeNoteFullName so each dependency type appears once. For each constructor parameter where `param.TypeNoteFullName` is not null, emit:
```
## Dependencies
- [[Sanitize(param.TypeNoteFullName)]] (`paramName`)
```
TypeNoteFullName is already the fully qualified name stored in ParameterInfo. This creates wikilink edges from class to its DI dependencies in Graph View.

7. **Properties and Fields** (STRC-05):
If there are properties or fields:
```
## Properties
- `PropertyName`: TypeName
- `FieldName`: TypeName
```
List properties first, then fields, each in declaration order. Use the TypeName from PropertyFieldInfo. Do not add wikilinks here (types may be external primitives); keep it clean.

8. **Members** section (compact navigation index, STRC-01):
```
## Members
- [[ClassName.MethodName]]

```
For each MethodId in typeInfo.MethodIds, extract the method name part and format as `[[{typeInfo.Name}.{methodName}]]`. Use declaration order (already sorted in TypeInfo). No access modifier prefixes per user decision. The wikilink format `[[ClassName.MethodName]]` must match the method note file name (which is `{ContainingTypeName}.{MethodName}.md` from the existing Sanitize logic).

The member index wikilinks need to use the same format that matches the file name. Extract the containing type name and method name from the MethodId or look up the MethodInfo from analysis.Methods to get ContainingTypeName and Name. Format: `[[ContainingTypeName.MethodName]]` where ContainingTypeName matches what is used in the file name.
