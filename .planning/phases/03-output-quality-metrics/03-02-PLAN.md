<task type="auto">
  <name>Task 1: Expand ObsidianEmitter with rich frontmatter, wikilink fix, danger callouts, and pattern detection</name>
  <files>Emission/ObsidianEmitter.cs</files>
  <action>
This is the largest task in Phase 3. The ObsidianEmitter needs significant changes across all three render methods plus the Calls/Called-by wikilink format.

**1. Add threshold fields and constructor to ObsidianEmitter:**
Add a constructor that accepts `int fanInThreshold = 10` and `int complexityThreshold = 15` and stores them as private readonly fields. The existing `EmitAsync` method stays the same signature (IEmitter interface unchanged).

**2. Add PatternDetection static helper:**
Add a private static method `DetectPattern(string typeName)` returning `string?`:
```
private static readonly (string Suffix, string Pattern)[] PatternSuffixes =
{
    ("Repository", "repository"),
    ("Controller", "controller"),
    ("Service", "service"),
    ("Middleware", "middleware"),
    ("Factory", "factory"),
    ("Handler", "handler"),
};
```
Check if `typeName` ends with each suffix (case-insensitive). Return the pattern name or null.

**3. Rewrite RenderMethodNote frontmatter:**
Replace the existing minimal frontmatter with expanded snake_case fields:
```yaml
---
namespace: "{method.Namespace}"
project: "{method.ProjectName}"
source_file: "{method.FilePath}"
access_modifier: "{method.AccessModifier}"
complexity: {method.CyclomaticComplexity}
fan_in: {fanIn}
fan_out: {fanOut}
tags:
  - method
  - danger/high-fan-in    (only if fanIn >= _fanInThreshold)
  - danger/high-complexity (only if complexity >= _complexityThreshold)
---
```
Compute `fanIn` and `fanOut` from `callGraph.GetCallers(method.Id).Count` and `callGraph.GetCallees(method.Id).Count`. Pass the full `AnalysisResult` to `RenderMethodNote` (not just `CallGraph`) since we need the call graph.

After frontmatter and header, add danger callout blocks BEFORE the method section:
```markdown
> [!danger] High Fan-In ({fanIn})
> This method is called by {fanIn} other methods. Changes here have wide impact.

> [!danger] High Complexity ({complexity})
> Cyclomatic complexity: {complexity}. Consider refactoring into smaller methods.
```
Only emit the callout(s) that apply. Add a blank line between callouts and between callouts and the next section.

**4. Fix wikilink format in RenderMethodSection (OUTP-01):**
In the Calls/Called-by sections, replace the `ExtractMethodName(callee)` call with a lookup that produces `ContainingTypeName.MethodName` format:
- If `analysis.Methods.TryGetValue(calleeId, out var calleeInfo)` succeeds, use `Sanitize($"{calleeInfo.ContainingTypeName}.{calleeInfo.Name}")`.
- Otherwise fall back to `ExtractMethodName(callee)` for methods outside the analysis (external).

This means `RenderMethodSection` also needs `AnalysisResult` as a parameter (not just `CallGraph`).

Also fix the self-link in the section header: change `#### [[{method.Name}]]` to `#### [[{Sanitize($"{method.ContainingTypeName}.{method.Name}")}]]` so it matches the method note file name.

**5. Rewrite RenderClassNote frontmatter:**
Replace the existing minimal frontmatter with expanded fields:
```yaml
---
namespace: "{typeInfo.Namespace}"
project: "{typeInfo.ProjectName}"
source_file: "{typeInfo.FilePath}"
access_modifier: "{typeInfo.AccessModifier}"
base_class: "{typeInfo.BaseClassFullName}"   # or ~ if null
interfaces:
  - "{iface}"
member_count: {typeInfo.MethodIds.Count}
dependency_count: {diDeps.Count}
pattern: "{detectedPattern}"               # or ~ if null
tags:
  - class
  - pattern/{detectedPattern}              # only if pattern detected
---
```
Call `DetectPattern(typeInfo.Name)` to get the pattern. Compute `diDeps` BEFORE frontmatter (existing code computes it for the Dependencies body section -- reuse that list).

**6. Rewrite RenderInterfaceNote frontmatter:**
Same base fields as class note, but:
- `member_count` = `typeInfo.MethodIds.Count`
- `dependency_count` = 0 (interfaces have no DI constructors)
- `pattern` = result of `DetectPattern(typeInfo.Name)` (interfaces can have patterns too, e.g., IRepository)
- Tags: `interface` base tag, plus `pattern/<name>` if detected

**7. Ensure consistent frontmatter across all note types:**
Common base fields in every note (method, class, interface): `namespace`, `project`, `source_file`, `access_modifier`, `tags`.
- Method notes add: `complexity`, `fan_in`, `fan_out`
- Class/interface notes add: `base_class`, `interfaces`, `member_count`, `dependency_count`, `pattern`

All numeric fields must be present (use 0, never omit). Tags must use bare strings (no `#` prefix).

**8. Update EmitAsync to pass AnalysisResult and thresholds to render methods:**
The `RenderMethodNote` method needs `AnalysisResult` (for call graph + method lookups). Update its signature to accept `AnalysisResult` instead of just `CallGraph`.

IMPORTANT ANTI-PATTERNS TO AVOID:
- Do NOT add `#` prefix to tags in YAML
- Do NOT omit metrics fields when value is 0
- Do NOT change the IEmitter interface (thresholds go on the constructor, not the interface)
- Do NOT compute fan-in during analysis (the graph is incomplete until all analyzers finish)
- For wikilinks, the `ContainingTypeName` is already fully qualified (e.g., "Code2Obsidian.Analysis.Analyzers.MethodAnalyzer"), so `[[Sanitize(ContainingTypeName.MethodName)]]` naturally handles namespace collisions
  </action>

**Program.cs** -- Add two new Option definitions alongside the existing `inputArgument` and `outputOption`:
```csharp
var fanInThresholdOption = new Option<int>("--fan-in-threshold")
{
    Description = "Fan-in threshold for danger tagging (default: 10)",
    DefaultValueFactory = _ => 10
};

var complexityThresholdOption = new Option<int>("--complexity-threshold")
{
    Description = "Cyclomatic complexity threshold for danger tagging (default: 15)",
    DefaultValueFactory = _ => 15
};
```

Add both options to the `RootCommand`:
```csharp
var rootCommand = new RootCommand("Analyze a C# solution and generate an Obsidian vault")
{
    inputArgument,
    outputOption,
    fanInThresholdOption,
    complexityThresholdOption
};
```

Update the `SetAction` handler to extract the new values:
```csharp
rootCommand.SetAction(async (parseResult, cancellationToken) =>
{
    var input = parseResult.GetValue(inputArgument)!;
    var output = parseResult.GetValue(outputOption);
    var fanInThreshold = parseResult.GetValue(fanInThresholdOption);
    var complexityThreshold = parseResult.GetValue(complexityThresholdOption);

    return await RunPipelineAsync(input, output, fanInThreshold, complexityThreshold, cancellationToken);
});
```

Update `RunPipelineAsync` signature to accept `int fanInThreshold` and `int complexityThreshold` parameters.

Update the emitter construction to pass thresholds:
```csharp
var emitter = new ObsidianEmitter(fanInThreshold, complexityThreshold);
```

This is the only wiring needed -- the emitter's constructor stores the thresholds and uses them in the render methods.
