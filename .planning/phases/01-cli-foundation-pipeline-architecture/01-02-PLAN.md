  <name>Task 1: Create Pipeline orchestrator, PipelineResult, and CliOptions</name>
  <files>
    Pipeline/Pipeline.cs
    Pipeline/PipelineResult.cs
    Cli/CliOptions.cs
  </files>
  <action>
    1. Create Cli/CliOptions.cs (namespace Code2Obsidian.Cli):
       - A sealed record: CliOptions(string SolutionPath, string OutputDirectory)
       - SolutionPath is the resolved absolute path to the .sln file
       - OutputDirectory is the resolved absolute path to the output vault folder
       - This record is created AFTER input resolution (not raw CLI args)

    2. Create Pipeline/PipelineResult.cs (namespace Code2Obsidian.Pipeline):
       - A sealed class (mutable during pipeline execution, read after):
         - TimeSpan AnalysisDuration, EnrichmentDuration, EmissionDuration
         - TimeSpan TotalDuration (computed: sum of all three)
         - int ProjectsAnalyzed, FilesAnalyzed, NotesGenerated, EnrichersRun
         - List<string> Warnings (accumulated from all stages)
         - bool HasFatalError
       - Method: int ExitCode => HasFatalError ? 2 : Warnings.Count > 0 ? 1 : 0

    3. Create Pipeline/Pipeline.cs (namespace Code2Obsidian.Pipeline):
       - A sealed class receiving IReadOnlyList<IAnalyzer>, IReadOnlyList<IEnricher>, IEmitter in constructor
       - Method: Task<PipelineResult> RunAsync(AnalysisContext context, string outputDir, IProgress<PipelineProgress>? progress, CancellationToken ct)
       - IProgress<PipelineProgress> is a simple callback interface for reporting progress to the UI layer. PipelineProgress is a record: (PipelineStage Stage, string Description, int Current, int Total) where PipelineStage is an enum { Loading, Analyzing, Enriching, Emitting }.
       - Define PipelineProgress and PipelineStage in the same file or a separate small file -- Claude's discretion.
       - Execution flow:
         a. Create AnalysisResultBuilder
         b. Run each IAnalyzer.AnalyzeAsync(context, builder, ct), reporting progress
         c. Build AnalysisResult from builder
         d. Record AnalysisDuration
         e. Create EnrichedResult wrapping AnalysisResult
         f. Run each IEnricher.EnrichAsync(analysis, enriched, ct)
         g. Record EnrichmentDuration
         h. Run IEmitter.EmitAsync(enriched, outputDir, ct)
         i. Record EmissionDuration, NotesGenerated from EmitResult
         j. Collect warnings from EmitResult into PipelineResult.Warnings
         k. Return PipelineResult
       - The Pipeline does NOT depend on Spectre.Console. Progress reporting is through the IProgress<PipelineProgress> abstraction so the Pipeline class remains testable without UI dependencies.
  </action>
  <verify>
    Run `dotnet build` from the project root. Must compile. Pipeline.cs must NOT contain any "using Spectre" imports.
  </verify>
  <done>
    Pipeline orchestrator exists with stage timing, progress callback, and clean separation from UI concerns. PipelineResult captures all data needed for the end-of-run summary. CliOptions provides typed options record.

  <name>Task 2: Rewrite Program.cs with System.CommandLine, Spectre.Console progress, and error UX</name>
  <files>

  <action>
    IMPORTANT: This task COMPLETELY REWRITES Program.cs. Delete ALL existing content and replace with the new implementation. The old logic has been ported to Analysis/ and Emission/ in Plan 01.

    1. System.CommandLine setup (per INFR-01, per research Pattern 1):
       - Create RootCommand with description "Analyze a C# solution and generate an Obsidian vault"
       - Positional Argument<string> "input" -- "Path to a .sln file or directory containing one"
       - Option<string?> "--output" with alias "-o" -- "Output directory for the Obsidian vault (default: <solution-dir>/vault/)"
       - Use async SetAction with (ParseResult, CancellationToken) signature
       - Call InvokeAsync() at the end (NOT Invoke() -- async action requires async invoke)
       - System.CommandLine auto-generates --help/-h/-? and --version

    2. Input path resolution (per user decisions, per research Pattern 2):
       - If input is a .sln file that exists: use it directly (Path.GetFullPath)
       - If input is a directory: find *.sln files inside
         - Exactly 1 found: use it
         - 0 found: error "No .sln file found in '<dir>'"
         - Multiple found: error listing them, ask user to specify
       - If input does not exist: generate "Did you mean?" suggestions
         - Look in the parent directory for *.sln files
         - List up to 3 closest matches (simple alphabetical listing is fine -- no need for Levenshtein in Phase 1, per research open question #3)
         - If no candidates: "File not found: '<input>'. No .sln files in '<dir>'."

    3. Output path resolution (per user decisions, per research Pitfall 5):
       - If --output provided: use it (Path.GetFullPath)
       - If not provided: default to <solution-dir>/vault/ (always relative to the resolved .sln file's directory, NOT the directory the user passed)

    4. Output path validation (per user decisions, per research code example):
       - Create output directory (Directory.CreateDirectory)
       - Write a test file (.code2obsidian-write-test), then delete it
       - On UnauthorizedAccessException or IOException: fail fast with clear error message
       - This happens BEFORE any analysis begins

    5. Spectre.Console progress integration (per INFR-02, per research Pattern 3):
       - Detect non-interactive terminal: check AnsiConsole.Profile.Capabilities.Interactive
       - If interactive: wrap pipeline execution in AnsiConsole.Progress().StartAsync() with columns: TaskDescriptionColumn, ProgressBarColumn, PercentageColumn, SpinnerColumn, ElapsedTimeColumn
       - Create progress tasks for each pipeline stage: "Analyzing...", "Enriching...", "Emitting..."
       - Adapt PipelineProgress callbacks to update Spectre.Console ProgressTask:
         - Update Description with current item (e.g., "Analyzing ProjectX...")
         - Increment progress counter
         - Set MaxValue when total becomes known
       - If non-interactive: still run the pipeline (progress bars display in simplified non-animated format automatically per Spectre.Console behavior -- no custom code needed)
       - CRITICAL (per research Pitfall 2): Do NOT call Console.WriteLine or AnsiConsole.MarkupLine while progress is active. Collect all warnings in PipelineResult and display AFTER progress completes.

    6. Pipeline composition (no DI container -- per research):
       - Create SolutionLoader, call LoadAsync to get AnalysisContext
       - Create analyzer list: [new MethodAnalyzer()] (or however the analyzers from Plan 01 are structured)
       - Create enricher list: [] (empty for Phase 1)
       - Create emitter: new ObsidianEmitter()
       - Create Pipeline with these components
       - Call Pipeline.RunAsync with the progress adapter and CancellationToken

    7. End-of-run summary (per user decisions, per research Pattern 5):
       - After pipeline completes and progress context closes, display summary using Spectre.Console Table
       - Table with columns: Stage, Duration, Items
       - Rows: Analysis (duration, "X projects, Y files"), Enrichment (duration, "skipped" for Phase 1), Emission (duration, "X notes"), Total (total duration, "X notes generated")
       - If warnings exist: display count and list each warning with a yellow "!" prefix
       - Use Spectre.Console markup for styling ([bold], [yellow], etc.)

    8. Exit codes (per user decisions):
       - 0: clean success (no warnings)
       - 1: success with warnings (some files skipped)
       - 2: fatal error (solution load failure, unwritable output, etc.)
       - Use PipelineResult.ExitCode for post-pipeline exit
       - For pre-pipeline errors (bad input, unwritable output): return 2 directly

    9. Error handling (per user decisions):
       - Wrap the entire RunPipelineAsync in try/catch
       - On InvalidOperationException (from path resolution): display error with Spectre.Console markup, return 2
       - On FileNotFoundException (from path resolution): display "Did you mean?" message, return 2
       - On any unexpected exception: display "Fatal error: <message>", return 2
       - Individual file failures during analysis/emission are handled inside those stages (skip + warn), not here

    Program.cs should be approximately 150-200 lines. Keep it clean and well-organized. Use #region or private static methods to section the code. The namespace is Code2Obsidian.
